# Ledger store keeps container nodes and leaf values disjoint

After typed nodes replaced `ArrayInOriginalData`, every slot inside the **Ledger** is either a `LedgerNode` (`LedgerObjectNode` or `LedgerArrayNode`) representing a container, or a recorded leaf value (primitive, Date, File, Map, Set, or `undefined` marking a removed key). The two are never mixed at the same depth: a slot is *one* of them, never a plain object that callers must duck-type. When the Proxy hands the **Ledger** a wholesale-captured **Baseline** value — e.g. assigning `data.contact = null` overwrites an entire prior object subtree, which `_setStoreValue` records as the leaf at that **Path** — the captured object/array is rewritten into a `LedgerNode` tree by `captureAsLedgerNode` before it lands in the store. Without that wrap, recorders and walkers would have to tolerate raw plain-object subtrees alongside `LedgerNode` containers, re-introducing the disambiguation tax that the typed protocol was meant to eliminate, and leaking the live-data shape into the **Ledger**'s storage rules. The replace-with-null-then-replace-with-partial-object test in `tests/tracked-instance.spec.ts` ("should display correct changedData after replace some value as object") is the regression that catches a "skip the wrap" optimization: without `captureAsLedgerNode`, the projection cannot reach the inner fields uniformly through `node.entries()`, so the partial-replace **Changed Data** comes out wrong. The disjointness is therefore a structural invariant of the store, not a performance choice — keep `captureAsLedgerNode` even when "just store the value as-is" looks like a tempting shortcut.
